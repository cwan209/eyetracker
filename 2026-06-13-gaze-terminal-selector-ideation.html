注视驱动的 macOS 终端选择器

日期 2026-06-13 主题 gaze-terminal-selector 聚焦 看哪个 terminal 就选哪个，不用鼠标 模式 elsewhere-software（全新项目）

看向某个终端，它就成为当前活动窗口——用眼睛代替鼠标来切换焦点。下面是 7 个排序后的方向，已对照现有产品、网络摄像头精度现实和 macOS 焦点 API 做过甄别。最大的发现：macOS 上没有任何已发布的工具能做到"注视→窗口聚焦"，而这个范式早已被验证（Vision Pro 的"看着即聚焦"），并且焦点切换是低频、粗粒度的——这让它远比"用注视当鼠标指针"更可行。

主题背景 · Topic Context

空白市场真实存在。macOS 上没有任何已发布工具能自动做"注视→窗口聚焦"。最接近的是 Tobii 的"注视高亮 Alt+Tab"——仅限 Windows、需要 Tobii 硬件、且仍需按一下键。苹果的"Eye Tracking"存在，但仅限 iOS/iPadOS，且不对第三方开放。Vision Pro 的"看着即聚焦 + 捏合确认"在架构上与本设想完全一致，验证了该范式。唯一的开源尝试是一个已废弃、仅 7 次提交的半成品。

精度决定设计。标准 MacBook 摄像头只有 RGB（没有深度——只有 iPhone/iPad 有 TrueDepth）。基于外观的网络摄像头注视估计，在 80 cm 距离、经校准后约为 3–4° / ~50 mm 误差——足以区分左右、象限，或宽于 ~150–200 mm 的窗格，但分不清狭窄的 tmux 分屏或 >4 列。专用 Tobii 硬件可达 ~0.5°。头部姿态（"看哪个显示器/哪一半"）比真正的注视容易得多。

为何可行。开发者每小时只切换焦点约 10–30 次——而鼠标指针有上千次微移动。你只需要粗粒度、偶尔的消歧，而非像素级精准指针。核心风险是 Midas 接触问题（看到什么就选中什么）；最有据可循的防御是"先看后按"（注视瞄准、按键触发）> 语音 > 长停留 > 眨眼。

macOS 给了你接口。CGWindowListCopyWindowInfo 读取窗口几何（无需权限）；AXUIElement 设置焦点（需要辅助功能权限）。AutoRaise（2025）是一个"焦点跟随鼠标"的守护进程——最直接的架构类比，把鼠标换成注视即可。一个坑：tmux/iTerm 的窗格不是 OS 窗口——选中一个窗格需要应用专属集成，或比网络摄像头更细的注视精度。

主题维度 · Topic Axes

  • A1 注视/注意力感知 — 输入信号的来源与方式（网络摄像头 CV、Tobii、iPhone Continuity Camera、头部姿态、混合）。
  • A2 选择与确认 — 把一瞥变成已提交的焦点切换；Midas 接触防御。
  • A3 目标解析 — 什么是可选的、注视如何映射到它（OS 窗口 vs 窗格 vs 显示器；死区、命中目标）。
  • A4 校准与精度 — 长时间保持准确；置信度、漂移、按显示器配置、优雅降级。
  • A5 工作流集成与反馈 — 融入开发者的日常；开关控制、焦点反馈、与键盘共存、超越终端的范围。

1.

先看后按 · 粗分区切换器

★ 首选 A2 · 选择与确认 置信度 85% 复杂度：中

描述可交付的 v1。摄像头（或头部姿态）估计你正看着哪个粗分区；你按一个 home 行组合键，把焦点提交到那个分区里的窗口。注视本身永不移动焦点——它瞄准目标，按键触发它。冷启动用头部姿态（"哪个显示器/哪一半"），零校准，置信度足够时再细化到注视。目标是按摄像头实际能命中的大小划分的分区，而非狭窄窗格。

依据直接："先看后按"是 HCI 研究中排名第一的 Midas 接触防御（注视+组合键 > 语音 > 停留 > 眨眼）；苹果 Vision Pro 正是以"看着即聚焦 + 捏合确认"发布了这一模式；macOS AXUIElement + CGWindowListCopyWindowInfo 提供焦点与几何；AutoRaise（2025）是一个可直接借鉴的"焦点跟随输入"守护进程。

理由这既是对你需求最直接、最诚实的回答（"看哪个终端就选哪个，不用鼠标"），又是最安全的设计：没有有意按键，焦点就不会改变；而粗糙的摄像头精度足够，因为你只在按键那一刻、在可见分区之间消歧。用现成 CV 和有文档的 macOS API，今天就能做。

缺点不是"纯免手魔法"——你仍要按一个键（但不用鼠标）。粗分区选不中狭窄的 tmux 分屏（见 #2、#4）。有些人想要零按键自动切换——作为更冒险的"勇敢模式"保留来做 A/B，而非默认。

2.

用 iPhone 当注视传感器（Continuity Camera）

A1 · 感知 置信度 55% 复杂度：大

描述与其和 Mac 那块平面 RGB 摄像头（~50 mm 误差）较劲，不如把你本来就放桌上的 iPhone 通过 Continuity Camera 用作 Mac 的摄像头——借用它的 TrueDepth 传感器和苹果已发布的眼动 ML。Mac 端 app 变成一个轻量的焦点执行器，消费手机算出的高质量注视流。

依据直接：Continuity Camera（Mac 用 iPhone 当摄像头）、iPhone TrueDepth、苹果 iOS 18 的 Eye Tracking ML 都是真实、已发布的能力。空白在于：没人把 iPhone 更强的注视感知接回来驱动 Mac 的窗口焦点。

理由在不买 Tobii 的前提下，这是最大的精度提升——有望达到窗格级分辨率，而且用的是大多数开发者已经拥有、已经立在桌上的硬件。它改变了下游"什么是可选的"（#4）。

缺点Continuity Camera 暴露的是视频流，不是注视 API——苹果把眼动数据保留在设备端私有，所以你很可能要自己在视频流上跑 CV，而非免费拿到苹果的注视。可能需要一个配套 iOS app，增加工作量。手机必须立稳并对准。

3.

用按键真值自我校准

A4 · 校准与精度 置信度 65% 复杂度：中

描述干掉校准仪式。每当你手动聚焦一个窗口（Cmd-Tab、点击、某个已知快捷键）且正看着它时，你就产生了一个免费的 (注视向量 → 已知窗口) 标签对。在正常工作中被动累积这些样本，持续拟合一个按显示器的注视→屏幕映射。第一分钟粗糙，午饭时就准了，零校准点——而且会自愈漂移（你往后靠、戴上眼镜）。

依据外部：从自然交互中做隐式/在线再校准，在网络摄像头注视研究中已成熟——WebGazer 的"点击即真值注视点"做点击再校准。这里的"点击"就是确认的焦点切换。

理由校准是头号采用杀手，也是网络摄像头注视"demo 能用、下午三点失灵"的原因。把每次确认切换变成训练数据，让精度曲线从衰减转为上升——这是让其他每个方向都能日常可用（而非只在 demo 里）的使能能力。

缺点冷启动：开头几分钟在样本累积前需要头部姿态兜底（与 #1 搭配）。切换时若看向别处会有标签噪声。按显示器布局做键控增加一点簿记。

4.

"抬高一级"：以项目/工作区为目标

A3 · 目标解析 置信度 70% 复杂度：中

描述别再把终端窗口当作单位。你真正在其间切换的是项目/仓库/tmux 会话/任务。把每个注视可分辨的区域映射到一整个工作区（项目 A=左显示器，项目 B=右），于是一瞥+组合键激活那个项目的全部上下文——终端、编辑器、分支、环境——而非单个窗口。粗注视完全够用：同时进行的项目只有 2–4 个，每个占一大块区域。

依据推理：瞄准精度必须 ≤ 目标的视角大小，否则选择不可靠。摄像头注视分辨 ~150–200 mm 区域；一个整项目区域正是这个尺寸，而 tmux 窗格约 40 mm。抬高目标海拔让粗注视自然契合，并彻底绕开"OS 窗口 vs 窗格"的问题——区域不在乎里面是什么。

理由用"往上走一级"而非"往下硬抠"来化解最难的技术分叉（窗格不是 OS 窗口、摄像头分不清），而且"免手切换我的整个项目上下文"比聚焦单个终端的工作流收益更大。

缺点需要定义什么是"工作区"并接好多应用激活（终端+编辑器+浏览器）——比纯窗口聚焦多。比字面需求更不精确。假设布局空间稳定（项目=固定区域）。

5.

注视即目的地（此处编写，发往那里）

A5 · 工作流与反馈 置信度 60% 复杂度：中

描述把"我在看哪里"和"谁拥有我的键盘"解耦。你一瞥的终端会悄悄成为你下一个派发载荷的目的地寄存器——粘贴的剪贴板、口述的命令、AI 的回答——而完全不从你正打字的地方抢走键盘焦点。在编辑器里编写，瞥一眼终端，按发送组合键，它就落到那里。cat error.log | gaze-pipe 会打进你正看着的那个终端。

依据推理：抢焦点是窗口管理里昂贵又易错的部分，而 Midas 接触之所以灾难性，正是因为它劫持了输入。如果注视设置的是一个非权威的目的地而非输入焦点，那么在你显式提交之前，看错一眼毫无代价——提交组合键才是真正的动作，注视只是预先寻址。粗精度对"这封信寄给三个盒子里的哪一个"完全够用。

理由通过让注视变成"建议性"，彻底绕开 Midas 接触，并解锁一个连鼠标都做不好的工作流（此处编写、发往那里）。这是一个真正全新的原语，而非 Cmd-Tab 的免手翻版。

缺点心智模型不同于"焦点跟着眼睛走"——可能不符合字面需求。需要清晰的反馈，让你信任载荷会落到哪。把输入路由进终端（而非 OS 焦点）需要按应用的粘贴/AX 管道。

6.

注视作为 AI 编码 agent 的上下文信号

A5 · 工作流与反馈 置信度 50% 复杂度：中

描述把"被注视的窗口/窗格/文件"作为可订阅信号暴露给 AI 编码 agent——一个 MCP "用户此刻在看什么？"资源，或一个本地端点。agent 把输出路由到你正看的窗格、在你正注视的终端里问澄清问题，并在你看向别处时暂停或延后长输出——等你看回来再恢复或总结。人的注意力成为 agent 循环的一等输入。

依据推理/外部：Vision Pro 确立了注视作为合法的低带宽意图通道；把它重构为 agent 输入（而非 UI 输入）是新颖之处，而 MCP 给了"把注意力暴露为资源"现成的传输。随着多个终端 agent 已普遍（Claude Code 等），"人正在关注哪个 agent"是一个真实、未填补的协调空白。

理由最高阶的复利押注：随着终端 AI agent 增多，"agent 知道你的注意力在哪"成为一个通用协调原语（输出路由、打断礼仪、注意力感知节奏），价值随每个新 agent 增长——把工具定位在持久的"人↔agent 注意力接口"，而非一次性切换器。

缺点依赖 agent 采用该信号（生态风险）——尽管你掌控自己的工具链。在 agent 集成之前，价值是推测性的。需要先有注意力流（#7）。

7.

注视注意力事件总线

A5 · 工作流与反馈 置信度 55% 复杂度：大

描述把"注视→聚焦"功能作为一个本地 IPC 服务的第一个消费者来发布，该服务发出结构化的"用户此刻在关注哪个窗口/区域/窗格"流（id、应用、置信度、注视时长、时间戳）。守护进程一次性拥有摄像头、校准和 Midas 接触过滤——这四个真正难的部分；其余一切（终端聚焦、通知门控、发往注视、按窗口勿扰、AI-agent 上下文）通过 Unix socket 订阅。确认方式（组合键/语音/停留）本身是读同一候选流的可插拔插件。

依据推理：构思过程浮现出 5+ 个不同的未来消费者，每一个若单独构建，都会各自重新解决摄像头采集、辅助功能权限、校准和 Midas 过滤。单生产者/多订阅者结构一次性付清这个代价，把第 N+1 个功能的边际成本压成一个轻量订阅者——这就是杠杆的定义。

理由把单一用途工具重构为平台原语——"macOS 从未发布的注意力 API"。焦点选择器成为总线的一个演示，而非天花板，且每个未来的注视点子都变成一个周末项目。

缺点经典 YAGNI 风险——在证明单一用例之前就造平台，为时过早。正确顺序几乎一定是：先做 MVP（#1），等第二个消费者真实存在时再抽取总线。把 v1 过度架构化可能拖垮项目。
