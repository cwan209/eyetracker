自适应注视焦点切换器（MVP）

title feat: Adaptive Gaze Focus Switcher (MVP) type feat date 2026-06-13 deepened 2026-06-13 origin docs/brainstorms/2026-06-13-gaze-terminal-focus-requirements.md

一个原生 macOS 菜单栏小工具:看哪个并排终端窗口、注视停留够久,键盘焦点就切过去——不用键、不用鼠标。架构是一个可无头测试的 GazeFocusCore SwiftPM 库(全部感知、决策、学习、生命周期逻辑)+ 一个薄 GazeFocus agent 壳(摄像头、Vision+CoreML 注视推断、Accessibility 窗口控制、菜单栏 UI)。安全兜底优先:任何不确定都按住焦点。整个工程由两个设备上的测量门控——注视精度,与「停下来读」的误切率。

Problem Frame · 问题背景

macOS 上没有工具按注视移动窗口焦点;在 2–3 个并排终端间切换要伸手摸鼠标或循环 Cmd-Tab。这活儿粗粒度、低频,普通摄像头够用——风险全在永不在错误时刻抢焦点:读日志时焦点跳走,接下来的键全打错窗口。计划把安全兜底闸链、打字闸、停留阈值下限、误切恢复当作承重工作。两个赌注在搭功能前先在设备上验证:(1) 摄像头到底能不能分清左/中/右;(2) 纯停留能不能在刻意的阅读停顿(停下打字去研究相邻窗口——打字闸覆盖不到的情况)里不误切。

Key Technical Decisions · 关键技术决策

KTD1. 双 target:可测试核心 + 薄 agent 壳。GazeFocusCore(无 AVFoundation/AppKit 导入)装全部纯逻辑;GazeFocus Xcode app 在 protocol 接缝后面包所有 OS 调用。沿用 personal_os 的 PersonalOSCore 约定。
KTD2. 注视 = Apple Vision 关键点 + CoreML 回归模型,跑在 ANE。VNDetectFaceLandmarksRequest(瞳孔+头姿)喂一个转换后的 L2CS-Net(MIT,~3.9° MAE)输出 yaw/pitch → 左/中/右分区。ARKit lookAtPoint(仅 iOS/TrueDepth)、MediaPipe(无原生 macOS Swift)均被否决。「摄像头到底能不能分区」是头号未验证赌注,由 kill-criterion 门控。
KTD3. 毕业时一次性 4 点校准。锚定未校准模型 5–15° 的个人偏差。solver 与逐点固视接受是纯核心逻辑。笔记本几何会漂移,所以校准是会话易变的——唤醒时重校验,不当作永久值。
KTD4. I-DT 固视检测(离散度 1.0°、100ms 窗),非 I-VT。F1 更高(0.96 vs 0.94)、对噪声更稳。停留计时前必须先确认稳定固视——靠这个离散度闸而非单纯时长区分一瞥与提交(固视时长是单峰的)。
KTD5. 学习目标 = 最小有意停留,按分位学习、受分布质量门控、叠在默认值上。阈值 = 有效停留事件的 80 分位(~100 样本有界缓冲),非均值、非眼→手延迟。毕业需 ≥40 有效事件或上限 且分布足够窄/单峰;太宽/多峰则钉在默认值并明说(origin R15 的退路)。默认 600ms、滑块可调、限定 [300,1000]ms。激活后被撤销的切换施加有界加长。训练/服务分布偏移列为设备验证项。
KTD6. 窗口控制 = CGWindowListCopyWindowInfo + Accessibility + 进程激活。几何/ownerPID/ownerName 无需 Screen Recording(只有标题才需要,从不读)。聚焦 = NSRunningApplication(pid).activate() 再 AXUIElementSetAttributeValue(kAXMainWindow/kAXFocusedWindow)+ kAXRaiseAction。光 raise 不转移键盘焦点,必须 activate。
KTD7. 两个 TCC 权限:Accessibility + Camera;v1 注视管线进程内。打字闸只读击键时刻(NSEvent .keyDown,搭 Accessibility,绝不读 .characters/修饰键——守卫不变量)。同一 monitor 检测两个硬编码 Cmd/Ctrl 组合键(暂停/撤销),无需第三方热键库。origin R29 的 XPC 隔离子进程延后:v1 进程内跑 Vision+CoreML(模型是离线转换的 MIT 静态产物、非运行时;二进制公证),残余风险接受,带再评估门。
KTD8. 非沙盒、Developer-ID 签名 + 公证;不上 Mac App Store。Accessibility 与沙盒不能共存;MAS 强制新 app 沙盒。硬约束。
KTD9. 实时采集路径:先用最简单的串行限定结构,尖刺证明必要才升级。U7 尖刺先评估串行队列限定对象,失败才上 actor+自定义 executor。nonisolated delegate 在跨边界前把 Sendable 的 CVPixelBuffer 从非 Sendable 的 CMSampleBuffer 里取出。头号技术风险。
KTD10. 静态加密 = FileVault + CryptoKit AES.GCM + Keychain 密钥;导出口令封装。macOS 上 NSFileProtectionComplete 无效,否决。拟合参数+少量近窗以 AES.GCM(AAD=bundle-id+schema 版本)封,密钥在 Keychain(绑签名身份 + WhenUnlockedThisDeviceOnly),文件在 per-user Application Support、排除备份。无密钥轮换(擦除重学)。导出口令封装、只含参数+校准、并警告这是个人行为数据。
KTD11. stopRunning() 真关相机。LED 硬连传感器电源,无软静音。暂停与睡眠/锁屏/挂起都 stopRunning(),切换去抖以避开 ~10s 硬件冻结。
KTD12. 安全兜底是脊梁:不确定就按住焦点。低置信、<2 个存活目标、全屏/Space 切换、唤醒后过时估计、Accessibility 丢失,全归结为「按住、不切」。Space 身份需私有 API,所以在 activeSpaceDidChangeNotification 上暂停。提交时「可见」是可判定谓词(当前 Space 内存活且未最小化);真遮挡用 kCGWindowLayer 近似、限制写明、完整遮挡检测延后。校准漂移产生「自信但错」的估计,兜底抓不住,故唤醒时重校验。
KTD13. 屏幕共享自动暂停延后;改发警告——但残余风险是第三方机密,不是用户烦扰。共享时一次注视切换可能把另一个终端的内容(密钥、他人数据)暴露给全场,由无意一瞥触发。可靠检测需 Screen Recording 权限 + macOS 反复弹窗。v1 发免权限警告、建议手动暂停;AE10 在 v1 明确未满足。会议进程启发式列为可能的后续部分方案。
KTD14. 相机采集在生命周期前端 opt-in(Idle 态),Collecting 明显读作「相机开」。授予相机权限不会启动采集;从 Idle 用户显式「开始收集」才启 AVCaptureSession,菜单全程明示相机在用。停留阈值滑块(默认 600ms)是独立于学习的手动覆盖,也是先证明核心赌注的途径。

Scope Boundaries · 范围边界

本期延后(计划内)

  • 屏幕共享自动暂停(R26/AE10)——v1 发免权限警告。再评估门:在向作者本机之外分发前重审——第三方机密暴露使「仅警告」只够个人单用户。
  • 免权限 XPC 隔离注视子进程(R29 加固)——v1 进程内(KTD7)。再评估门:在分发给作者之外的任何用户、或采集路径引入任何非内置运行时依赖前重审。
  • 眼→手延迟信号——暂存;v1 学最小有意停留(KTD5);origin R13/F4/AE7 重释为有效停留过滤。
  • 完整遮挡检测——提交时「可见」用近似(当前 Space 内存活未最小化);窗后遮挡延后(KTD12)。
  • 会议进程屏幕共享启发式、激活后撤销增益调参(超出有界 nudge)、GazeIntent 式 ML 意图分类器、私有 SkyLight focus-without-raise——均为后续增强。

更久之后(来自 origin)

多显示器一等公民、tmux/iTerm 窗格级、泛化到任意 app、可见停留进度环、阅读 vs 瞄准扫视路径分类、iPhone Continuity Camera 作高精度传感器。

本期范围内的退路(来自 origin)

纯停留若过不了任一设备门(注视精度或阅读停顿误切率),「先看后按」键确认是退路——预备不预建。

考虑过但未选

按需/空闲门控采集(仅靠瞬时触发或键盘空闲后开相机)会一举缩小常开隐私面、LED 常亮成本与进程内爆炸半径,但牺牲用户选择的「无痕常守」手感。v1 故意选常开;在此记录以使取舍显式、非默认继承。

本产品身份之外(来自 origin)

注视事件总线、注视即目的地、注视作 AI agent 上下文、注意力门控通知路由、注意力分析——独立产品。

Risks & Dependencies · 风险与依赖

  • 注视精度 kill-criterion(硬门,U7 尖刺 2)。Phase-2 搭功能前预注册线:在最低规格内置摄像头、≥5 用户、≥2 光照、戴/不戴眼镜、校准后 ≥95% 正确分区。相关量是真实笔记本上的分区错误率,非模型 3.9° 基准 MAE。不达线:停下,走「先看后按」退路。
  • 静默阅读停顿误切率(硬门,U7 尖刺 3)——纯停留赌注的真正考验。打字闸只在击键近窗内生效;刻意的阅读停顿(停下打字去研究相邻日志)是持续可信固视,过所有闸、与提交不可区分。R12 的一瞥/提交分离覆盖不到。测量持续阅读时的误切率;不可接受则这一项就逼出「先看后按」退路。比原始精度更可能成为门。
  • Swift 6.2 实时路径并发。先尖刺、最简结构优先(KTD9);@preconcurrency 是最后手段。
  • 学习收敛。80 分位是否收敛、是否「无痕」未证;分布质量门拒绝拟合宽/多峰分布,默认值+滑块使错值非灾难,撤销率信号标记净负学习器。
  • 笔记本校准漂移产生兜底抓不住的「自信但错」估计(兜底抓低置信非偏差)——靠唤醒重校验缓解,不当永久值。
  • 进程内注视管线残余风险(KTD7)——v1 接受;XPC 隔离延后带再评估门。
  • 屏幕共享数据暴露是第三方机密非用户烦扰(KTD13)——R26 落地前仅靠 v1 警告+手动暂停。
  • AXIsProcessTrusted 陈旧缓存——无撤销通知;U8 健康监控缓解。
  • 供应链。无第三方 Swift 运行时依赖(去掉了 KeyboardShortcuts);模型产物构建期校 hash(U6);提交 Package.resolved;v1 无自动更新器。
  • 依赖:AVFoundation、Vision、CoreML、ApplicationServices/AppKit(全 Apple 原生);coremltools 7+(离线、模型准备);L2CS-Net MIT 权重。无第三方 Swift 包。

Open Questions · 待决(留给实现)

  • 左/中/右 yaw 边界与中区死区宽——设备上对校准后模型调。
  • 模型变体:ResNet-50(~90MB/~3.9°)vs MobileOne-S0(~5MB/更高误差)——测真实摄像头精度后选。
  • 眼区 ROI(Vision 瞳孔)vs 头姿归一全脸作输入——验瞳孔关键点稳定性(U7)。
  • 恢复窗 N 与 R17 撤销 nudge 增益——观察真实撤销后调。
  • 毕业数量(≥40 起点)、收集上限时长、「可用」的分布质量阈。
  • 阅读停顿误切率(U7 尖刺 3)——纯停留过线,还是这一项逼出「先看后按」退路？
  • 校准漂移容忍——偏移多陈旧才产生自信错切,唤醒重校验是否需要更细的漂移检测器(如关键点尺度变化)。
