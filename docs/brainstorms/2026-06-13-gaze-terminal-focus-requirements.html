自适应注视焦点切换器（MVP）

date 2026-06-13 topic gaze-terminal-focus

一个 macOS 菜单栏小工具：用摄像头追踪视线，当你的目光在某个并排的终端窗口上停留足够久，就把键盘焦点切过去——不用键、不用鼠标。只要它对目标不确定（你在打字、置信度低、目标有歧义），就按住焦点不动；同时从你平时手动切窗口的行为里学你个人的「看了多久才动手」节奏，让切换逐渐变得无痕。

Problem Frame · 问题背景

macOS 上没有任何工具按「你看哪里」来移动窗口焦点。在 2–3 个并排终端之间切换，意味着伸手摸鼠标或在心流中循环 Cmd-Tab——对一个本来全靠键盘的开发者是一份小而持续的税。苹果的 Eye Tracking 只在 iOS/iPadOS 上、且不对第三方开放；Vision Pro 的「看着即聚焦 + 捏合」验证了范式，但在 Mac 上无处可寻。

这个机会真实，是因为这活儿粗粒度且低频：开发者每小时切焦点是几十次量级，不是上千次，而且只需要分辨两三个很宽的窗口。粗注视（左/中/右）对这么宽的窗口足以越过摄像头 ~50mm 的误差。难点从来不是感知——而是做一个永远不在错误时刻抢焦点的免手切换。那个失败（你读日志时焦点跳走，接下来的键全打错窗口）正是会让人第一天就卸载的原因，也是整个 MVP 的设计核心。贯穿始终的总原则是：一旦不确定，就不切。

Key Decisions · 关键决策

纯停留、不按键——一个刻意的、被防御过的赌注。 较高风险的赌注 视线在窗口上停留超过个人阈值就提交焦点，不用组合键、点击或眨眼。这是对原始需求最「魔法」的解读，也是你明确的选择。它也是较冒险的那个：上游 ideation 把「先看后按」(注视瞄准、按键提交) 排为更安全的默认，把零确认停留只留作「勇敢模式」。本文档有意下这个勇敢模式的赌注，但靠四件事一起防御——打字闸 (R7)、把阈值设在「随意一瞥」之上 (R12)、通用安全兜底 (R8)、以及误切恢复手段 (R10)。退路：若「零抢焦点」成功标准在测试中不达标，「先看后按」键确认模式是本期范围内的退路（是预案，不只是延后）。

真眼球注视，不是头部姿态。 单屏并排的两个终端只差几度，你几乎不转头，是在转眼睛。头部姿态（更易、免校准）分不清同屏窗口，所以 MVP 必须用真正的注视估计。

窗口无关的目标解析；「终端」是验证焦点，不是能力上限。 工具把视线映射到它落入的那个窗口矩形。解析逻辑里没有任何终端专属的东西，所以 v1 对任意标准窗口都成立，不会去建以后泛化时还得拆掉的终端检测耦合。把验证范围收在并排终端上让测试诚实；同时这意味着只读窗口几何（不读标题或应用身份），从而不需要 Screen Recording 权限。

个人「看了多久才动手」延迟是无痕的机制——有界，不无限。 眼睛领先手：动作之前视线先落到目标上。「视线到达窗口」到「随后手动操作」之间的间隔，是这个用户「我已经决定了」那一刻的零标注度量，阈值由它导出 (R11)，于是自动切换正好卡在你本来也会动手的时刻。两个诚实的限制塑造了设计：这个间隔的大小与情境相关（预先决定的切换又快又紧；反应式的「瞥一眼再决定」又慢又散），所以阈值用稳健统计量、且放在「随意一瞥」之上 (R12)，而非原始均值；而这个信号在鼠标驱动的收集期最丰富、转纯注视后变弱，所以模型在收集期收敛、之后稳定，而不是永远复利 (R17)。

渐进冷启动，有界到永远不会把你卡住。 首次运行只观察、不切任何东西 (R14)。学到把握后弹一次提示问是否启用注视模式；若在有界时间内达不到把握（或你的延迟分布太宽给不出可信阈值），它会提议用一个稳妥默认值毕业，而不是无限等待；你也可以随时从菜单手动启用 (R15, R16)。观察期基本无感，因为信号来自你本来就在产生的行为。

全程本地，模型越用越准——定位为质量收益，不是护城河。 已纠偏 所有注视推断与学习都在本地，数据不出 Mac (R27)。模型越来越贴合你是一种 UX 质量收益，不是锁定/留存武器——对一个免费、本地、单用户工具，那种说法是空的，还会把设计推向「故意不让模型可移植」这种伤害用户的方向。所以模型设计上可导出 (R28)。真正持久的优势是市场空白，不是重学成本。

安全兜底是脊梁。 每一个不确定的条件都归结到「按住当前焦点、什么都不切」(R8)。这一条规则吸收了：注视置信度低、目标有歧义、目标在停留途中消失、唤醒后估计过时、权限丢失、全屏/单窗口的 Space、以及第二块屏幕出现——让每一种都成为已定义的安全降级，而非未定义行为。

Key Flows · 关键流程

F1. 冷启动收集与有界毕业
  • 触发 首次启动（权限已授予）或模型重置。
  • 步骤 观察视线 + 手动切焦点动作，采集「看-动手」样本，不切任何东西，显示收集进度。够把握时弹一次提示启用；若先到收集上限，则提议用稳妥默认值毕业。
  • 结果 启用 → Active；拒绝或还没提示 → 留在 Collecting/Ready，菜单常驻启用入口。
  • 覆盖 R11, R12, R13, R14, R15, R16, R22
F2. Active 停留切换（happy path，安全兜底门控）
  • 触发 Active 下视线落到非当前窗口。
  • 步骤 滤掉扫视；仅当置信度够且 ≥2 个存活目标、用户未在打字、停留超过个人阈值、且提交时目标仍存活可见时，raise + 设键盘焦点并显示切换指示。
  • 结果 被注视窗口激活；切换伴随可察觉信号与撤销手段。
  • 覆盖 R4, R5, R6, R8, R9, R10, R21
F3. 打字闸抑制
  • 触发 当前窗口最近有按键。
  • 步骤 打字期间抑制切换；停顿后重新武装。菜单反映「武装/抑制」。
  • 结果 任务中读另一窗口绝不抢焦点。
  • 覆盖 R7, R20
F4. 被动学习采样
  • 触发 视线到达后随即有手动切焦点动作。
  • 步骤 仅当手动动作落在被注视的同一窗口才记录「视线→动作」间隔与映射，否则丢弃。激活后仍低量采集以刷新样本。
  • 结果 个人阈值与校准在收集期收敛、之后保持刷新。
  • 覆盖 R11, R12, R13, R17
F5. 误切恢复
  • 触发 一次你不想要的自动切换。
  • 步骤 切换给出可察觉、有时限的信号；用户在 N 秒内触发撤销动作恢复原焦点。被撤销的切换作为负学习信号回灌。
  • 结果 焦点归位；阈值随时间加长。
  • 覆盖 R10, R17, R21
F6. 暂停、挂起与降级
  • 触发 用户暂停；或系统睡眠/锁屏/登出；或开始录屏/共享；或注视跟踪丢失；或权限被撤销。
  • 步骤 切换停止、相机停采（手动暂停、睡眠/锁屏、录屏/共享）；跟踪丢失或权限丢失降到明确的非激活态。唤醒/恢复后首次切换前需要新的可信注视。
  • 结果 焦点不动；绝不在过时或低置信估计上切换；菜单始终显示原因。
  • 覆盖 R8, R18, R19, R23, R24, R25, R26

Requirements · 需求

感知与目标解析
ID需求
R1.工具用 Mac 摄像头估计用户的屏上注视点，精度足以在正常坐姿距离下分辨 2–3 个并排窗口。
R2.可选目标是实际的屏上窗口矩形，实时映射；视线归属到包含它的那个窗口。目标解析只读窗口几何（不读标题/应用身份），且窗口无关——v1 以终端为验证对象，但不建终端专属过滤。
R3.遇到歧义时不切、而非猜：视线落在歧义区（靠近共享边界或窗口重叠处）、窗口多到注视无法可靠区分、或窗口太小无法分辨时，一律不切。
选择与安全（永不在错误时刻抢焦点）
ID需求
R4.Active 态下，焦点仅靠注视停留提交——不需要键、点击、眨眼或语音。
R5.滤除途中的眼动（扫视）；只有稳定固视才能开始计停留。
R6.切换仅在对不同于当前窗口的持续、可信固视上触发；当前窗口零成本保持焦点（粘滞）。对已聚焦窗口的停留是 no-op——不调用聚焦、不重触发切换指示。
R7.当用户正在当前窗口打字时，注视切换被抑制，停顿片刻后才重新武装。
R8.目标不确定时一律安全兜底为粘滞焦点：注视置信度低、未检测到人脸/眼睛、当前 Space 上存活目标少于两个、唤醒后估计过时、或权限丢失——都归结到「按住当前焦点、什么都不切」，并在菜单态反映。
R9.提交前目标重校验：设焦点时目标必须存活可见；若它在停留途中关闭、退出、最小化或被遮挡，则中止（no-op）。停留途中新出现在视线下的窗口不会追溯性满足进行中的停留。
R10.恢复：每次自动切换都给出可察觉、有时限的「发生了切换」信号，并有明确的撤销动作恢复原焦点（如暂停热键在 N 秒内撤销最近一次切换，或专用 undo）。
学习与冷启动
ID需求
R11.工具测量用户「视线到达 → 手动动作」的延迟，并用它的稳健统计量（而非原始均值）导出停留阈值。
R12.阈值把有意提交与随意阅读一瞥分开：学习「提交延迟」与「随意一瞥停留」两个分布，把阈值放在一瞥范围之上，使非打字的阅读停顿不触发切换。
R13.只有当手动动作落在被注视的同一窗口，样本才计数；无关的鼠标移动丢弃。
R14.首次运行进入 Collecting 态，只观察、不切任何东西。
R15.收集有界：达到把握时弹一次提示启用；若在有界时间内达不到把握、或延迟分布太宽/多峰给不出可信阈值，工具提议用稳妥默认值毕业，而非把用户卡住。用户也可随时从菜单手动启用。
R16.Ready 态：工具持续观察，菜单常驻一个清晰带标签、与提示同措辞的「启用注视模式」入口。仅在用户同意后激活；拒绝过提示的用户因菜单入口常驻而不会被卡住。
R17.激活后用更粗的可用信号继续适应：用户在 N 秒内撤销的切换是负信号（加长阈值），偶发的手动切换被重新挖成新的延迟样本。模型被理解为收集期收敛、之后稳定，而非无限锐化。
控制、状态与反馈
ID需求
R18.全局热键 + 菜单栏开关随时暂停/恢复注视切换。
R19.暂停/关闭态下，相机采集停止——不只是软件静音。
R20.菜单栏显示当前状态，至少区分：collecting、ready、active-武装、active-被打字闸抑制、tracking-lost、paused。
R21.切换时，被聚焦的新窗口由工具自带的补充指示明确标出，而非只靠 macOS 原生焦点外观（深色/透明终端主题常把它压掉）。
R22.收集期间，工具显示距离「就绪」的进度，让用户知道还差多少能毕业。
R23.在 Accessibility 与 Camera 权限授予前，工具显示清晰的非激活态并引导授权；绝不静默失败。
系统事件
ID需求
R24.显示器睡眠、锁屏或登出时，相机停采、切换挂起；唤醒后首次切换前需要新的可信注视，绝不在过时估计上抢焦点。
R25.运行时环境变化安全降级：接入第二块屏 → 跟踪限定到主屏、忽略其它屏窗口；全屏或目标少于两个的 Space → 切换失效（inert）；运行时权限被撤销 → 明确的禁用态。
R26.当录屏或屏幕共享会话进行中，工具自动暂停切换（或显示常驻警告），使焦点变化不会把另一个窗口的内容泄露给观看者。
隐私与信任
ID需求
R27.所有注视推断与学习数据留在本地；图像或注视数据不出 Mac，并遵守 macOS 相机使用指示灯。
R28.存储的模型与样本视为敏感行为数据：放在沙盒容器、排除出云/Time Machine 备份、静态加密。原始样本在模型处理后删除（或在有界保留窗口内删除），长期只保留导出的阈值/校准参数。模型可由用户导出。
R29.Accessibility 授权是高价值信任边界：工具只申请所需的最小 API 面，把任何第三方注视估计组件跑在没有 Accessibility 权限的沙盒子进程里，且只改窗口焦点——绝不注入合成键鼠事件。

Acceptance Examples · 验收示例

AE1. 覆盖 F2 / R4, R6
给定 注视模式激活、焦点在左终端，当 用户在右终端保持稳定固视超过个人停留阈值，则 焦点移到右终端。
AE2. 覆盖 F3 / R7
给定 用户正在左终端打字，当 他瞥右终端读一秒日志，则 焦点留在左终端，按键继续落在那里。
AE3. 覆盖 F2 / R5, R12
给定 注视模式激活，当 用户在阅读停顿中扫过或短暂停在右终端但未超提交阈值，则 不发生切换。
AE4. 覆盖 F2 / R9
给定 用户已在右终端停留，当 该窗口在提交前一刻关闭或最小化，则 切换中止（no-op），焦点留在左终端。
AE5. 覆盖 F2 / R6
给定 焦点在左终端，当 用户视线停在同一个左终端，则 什么都不发生——不重聚焦、不出切换指示。
AE6. 覆盖 F2 / R8
给定 注视模式激活，当 摄像头被遮挡或用户看向屏幕之外，则 工具按住当前焦点、不切，并显示 tracking-lost 状态。
AE7. 覆盖 F4 / R13
给定 任意状态，当 用户视线停在右终端、随后点进左终端，则 不记录学习样本。
AE8. 覆盖 F1 / R14, R15
给定 首次运行，当 用户正常工作，则 在模型达到把握并被用户接受、或用户从菜单启用之前，不发生任何焦点切换。
AE9. 覆盖 F5 / R10
给定 一次用户不想要的自动切换，当 他在恢复窗口内触发撤销，则 原焦点恢复，且该事件作为负学习信号回灌。
AE10. 覆盖 F6 / R26
给定 注视模式激活，当 录屏或屏幕共享会话开始，则 切换自动暂停（或出现常驻警告）。

Success Criteria · 成功标准

  • 打字零抢焦点：测试中，当前窗口的连续打字期间不触发任何自动切换。
  • 阅读停顿不误切：阈值落在用户随意一瞥之上，模糊或不确定的注视产生「不切」而非误切。
  • 在可信固视上可靠分辨 2–3 个并排窗口。
  • 「无痕」手感：收集后，自动切换卡在用户自然的「看-动手」时刻——主观上不抢跑也不迟钝。
  • 有界 onboarding：用户在一个现实会话内达到可用的 Active 态——通过有把握毕业、收集上限默认值、或手动启用——绝不被卡在 Collecting。
  • 可恢复：误切可察觉、一个动作可撤销，且用户总能通过工具自带指示分辨焦点在哪。
  • 信任控制：暂停/恢复即时且明显；在跟踪丢失、睡眠、屏幕共享或权限撤销时降级到明确的安全态。

Scope Boundaries · 范围边界

本期延后

  • 多显示器、笔记本 + 外接屏（v1 限定主屏）。
  • tmux / iTerm 窗格级选择（子窗口目标对摄像头精度太窄）。
  • 泛化到任意应用或工作区/项目区域（机制已允许，v1 只是不验证它）。
  • 可见的停留进度环 + 看走取消（用打字闸 + 阈值高于一瞥 + 恢复手段替代）。
  • 超出打字闸与一瞥/提交阈值的「阅读 vs 瞄准」扫视路径分类。
  • iPhone Continuity Camera（TrueDepth）作为更高精度传感器。

本期范围内的退路（非延后）

  • 若纯停留在测试中没通过「零抢焦点」标准，「先看后按」键确认模式是退路。它作为预案、不预先构建，但明确在 MVP 的风险范围之内。

本产品身份之外

这些是 ideation 里的独立产品，不是焦点切换器的 v1 特性：通用的注视「注意力事件总线」、注视即目的地（「把这个发到我正看的终端」）、注视作为 AI 编码 agent 的上下文、注意力门控的通知路由、以及注意力分析/热力图。

Dependencies / Assumptions · 依赖与假设

  • 必需 macOS Accessibility 权限（设窗口焦点）与 Camera 权限（注视估计）；两者授予前工具不激活。不需要 Screen Recording 权限，因为目标解析只读窗口几何、不读标题/应用身份（R2）。
  • Accessibility 授权很宽——能读改任意窗口的焦点状态——所以被当作高价值信任边界（R29）。第三方注视库的被攻破不得借此转为键盘记录或事件注入。
  • 摄像头注视估计经校准后，在正常坐姿距离能可靠分辨左/中/右。（假设——由 `ce-plan` 验证估计方案与可达精度。）
  • 眼睛领先手：视线先于手动动作，所以「视线起点 → 动作起点」间隔是有效的个人信号。其方向已被充分确立；其大小与情境相关，所以阈值取稳健统计量并放在随意一瞥之上（R11, R12）。
  • 用户在冷启动期产生足够的手动切焦点动作来训练模型；收集上限 + 默认毕业（R15）界定了「产生不够」的风险。
  • v1 面向单台 Apple Silicon Mac、单用户、单显示器。

Outstanding Questions · 待决问题

规划前需解决

无阻塞项。纯停留 vs 键确认的赌注已作为带退路的刻意决策记录在案（见关键决策）；用户已确认纯停留。

留给规划

  • 注视估计方案与库，以及校准 UX（显式快速校准 vs 纯被动自校准）。
  • 收集上限值与触发毕业提示的「足够把握」判据——由「一个典型开发会话产生多少合格样本」的估计来界定。
  • 如何把「随意一瞥」分布与「提交延迟」分布分开，以及个性化前用的默认停留时长。
  • 恢复窗口长度 N（秒），以及撤销动作是暂停热键还是专用 undo。
  • 拒绝提示后的确切再提示策略（提示节奏 vs 仅靠菜单入口）。
  • 「干净样本」的精确定义（空间容差与时间窗），以及「正在打字」的定义（按键时近窗）。
  • 重叠窗口的聚焦语义——raise-and-focus 是默认（F2, R21）；开放的只是「目标完全被另一窗口挡住」时怎么办。
  • 存储模型/样本的静态加密机制与备份排除方式。
  • 录屏/共享的检测机制，以及响应是自动暂停还是常驻警告。
